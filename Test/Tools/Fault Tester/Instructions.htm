Fault Tester Options
 
Attached is a copy of the fault tester source code named fault_tester.zip, also I have included a precompiled version called fault_tester_exec.zip. I added a boolean variable in the code that you can set to TRUE or FALSE called bErrorVerbose. Setting this to TRUE causes ERROR codes to be displayed in the output. I had to do this for some functions to determine why they were failing. The compiled version has verbose enabled.
 
 
Using the Tester
 
To use the tester, intercept it using Holodeck. Once the menu is displayed, check the fault in Holodeck that you wish to test. Then, enter the corresponding menu item in the tester and press enter. The results of the test will be displayed.
 
 
Implementation Issues
 
I ran into some interesting issues while implementing the tester. The most notable ones were:
 
1) No Output To Console Window
2) Crashes When Injecting Memory Faults
 
To get around these issues, I created my own print function and used the exported disableInterceptionInCurrentThread() and enableInterceptionInCurrentThread() functions of HEATDLL to disable interception before a print operation is performed. For the memory function tests, I had the same problem where I needed to allocate memory but could not because of the fault. So, I just surrounded the allocation with those two functions to allow it to occur, then the function test could use that successfully allocated memory to test the function. To get those functions to work, I basically got the function pointer to them. If the function pointer cannot be obtained, the functions are ignored (this way you can run the fault_tester without needing Holodeck just to verify that stuff fails when Holodeck doesn't inject faults).
 
 
Results from Using the Tester
 
The result is that MoveMemory and CopyMemory, of which either one or both are used in every memory fault, are not being intercepted. This can be verified by disassembling a normal MoveMemory/CopyMemory and a supposedly intercepted MoveMemory/CopyMemory. According to documentation online, MoveMemory and CopyMemory are essentially the same functions which call RtlMoveMemory in KERNEL32.DLL... So, here is the disassembly of RtlMoveMemory before being intercepted:
 
0:000> u kernel32!RtlMoveMemory
ntdll!RtlMoveMemory:
77f82d8c 56               push    esi
77f82d8d 57               push    edi
77f82d8e 8b742410         mov     esi,[esp+0x10]
77f82d92 8b7c240c         mov     edi,[esp+0xc]
77f82d96 8b4c2414         mov     ecx,[esp+0x14]
77f82d9a fc               cld
77f82d9b 3bf7             cmp     esi,edi
77f82d9d 761a             jbe     ntdll!RtlMoveMemory+0x2d (77f82db9)
 
And, here is the disassmebly after being intercepted:
 
ntdll!RtlMoveMemory:
77f82d8c 56               push    esi
77f82d8d 57               push    edi
77f82d8e 8b742410         mov     esi,[esp+0x10]
77f82d92 8b7c240c         mov     edi,[esp+0xc]
77f82d96 8b4c2414         mov     ecx,[esp+0x14]
77f82d9a fc               cld
77f82d9b 3bf7             cmp     esi,edi
77f82d9d 761a             jbe     ntdll!RtlMoveMemory+0x2d (77f82db9)
 
There should be a JMP at/near the beginning of the intercepted one, but there isn't. So, they are not intercepting this function from what I can tell.
 
Other than that, I had no problems, at least in verifying that the functions were setting the error codes and return values that had been specified in the fault_db.xml file.
 
 
 
Future Uses
 
I am hoping someone can modify the fault tester sometime in the future to easily read the fault_db.xml file and automagically generate the menus and perform the checks. This will be useful for regression testing later.
